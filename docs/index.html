01Architecture du dépôt

Deux composants distincts, un même dépôt, une seule chaîne de confiance.

🖥️

/frontend — SPA statique

HTML/CSS/JS moderne, sans backend embarqué. Consomme l'API distante et sert de banc de diagnostic. Publiée sur GitHub Pages.

⚙️

/backend — API REST

Node.js / Express, données sensibles, clés d'API et accès externes. Conteneurisée, scannée puis déployée sur Vercel.

02Gouvernance Git & branches

La politique de sécurité est lisible directement dans le YAML — un auditeur constate la stratégie de promotion sans exécuter le pipeline.

BrancheRôleRègles
stagingPivot d'intégrationCI complète sur chaque push & PR
mainProduction stablePush directs interdits · PR review · status checks stricts · env production
preprodPréproduction / vitrinePublie cette GitHub Page de présentation
💡 Tip — visibilité de la protection

Le déclencheur cible staging et main. Les blocs if: github.ref == 'refs/heads/main' combinés à un needs strict prouvent qu'un déploiement sur main exige le succès absolu des jobs de validation en amont.

.github/workflows/ci-cd.yml
on:
  push:
    branches: [staging, main]
  pull_request:
    branches: [staging, main]

# Moindre privilège : lecture seule par défaut, aucun héritage d'écriture
permissions:
  contents: read

concurrency:
  group: ci-cd-${{ github.ref }}
  cancel-in-progress: true

03Durcissement local — Shift-Left

Avant qu'une seule ligne n'atteigne GitHub, le hook pre-commit garantit l'intégrité du poste de travail.

✅

actionlint

Validation de l'ensemble des fichiers .github/workflows/.

🔎

gitleaks staged

Scan ciblé uniquement sur les modifications indexées.

🚫

Contrôle de structure

Tout .env, .pem ou .key dans l'index avorte le commit.

.git/hooks/pre-commit — garde-fou fichiers sensibles
if git diff --cached --name-only | grep -E '\.(env|pem|key)$'; then
  # message d'erreur rouge explicite, opération annulée
  echo -e "\e[31mSécurité : Tentative de commit d'un fichier de
  configuration ou d'une clé en clair. Opération annulée.\e[0m"
  exit 1
fi

Règle Gitleaks sur-mesure

Une expression régulière personnalisée intercepte les jetons internes SECWALLET_ suivis d'exactement 24 caractères alphanumériques majuscules, renforcée par un seuil d'entropie.

gitleaks.toml
[[rules]]
id = "secwallet-internal-token"
regex = '''SECWALLET_[A-Z0-9]{24}'''
entropy = 3.5
keywords = ["SECWALLET_"]

04Gestion des secrets par enveloppe

Aucun secret de production en clair, mais des structures YAML auditables pour des git diff propres.

  1. Paire de clés asymétriques générée avec age — la clé privée est ops.txt (ignorée par Git).
  2. .github/secrets-prod.yaml contient les clés applicatives (DATABASE_URL, JWT_SECRET, API_KEY).
  3. Chiffrement SOPS ciblé : seules les valeurs deviennent des blocs ENC[...], les clés structurelles restent lisibles.
chiffrement sélectif par regex
# 1) génération de la paire de clés age
age-keygen -o ops.txt

# 2) chiffrement des seules valeurs applicatives
sops --age "$(grep -o 'age1.*' ops.txt)" \
     --encrypt --encrypted-regex '^(DATABASE_URL|JWT_SECRET|API_KEY)$' \
     .github/secrets-prod.yaml
🔐 Déchiffrement au runtime

En CD, la clé privée provient du secret GitHub SOPS_AGE_KEY. Le déchiffrement se fait en mémoire RAM et les valeurs sont injectées à la volée dans la commande Vercel — jamais écrites sur le disque du runner.

05Conteneurisation multi-stage & GHCR

Image serveur minimale, non-root, filtrée par chemin, scannée avant toute publication.

🧱

Multi-stage & non-root

Étape deps (npm ci --omit=dev) puis runtime Alpine, utilisateur nodeapp dédié.

🎯

Filtrage de chemins

dorny/paths-filter ne (re)build l'image que si backend/** a changé.

🛡️

Scan pré-publication

Trivy en mode image, --exit-code 1 --severity HIGH,CRITICAL.

🚀

Push conditionnel

Publication sur GHCR taggée au SHA uniquement si le scan réussit, sur main.

backend/Dockerfile
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev

FROM node:20-alpine AS runtime
ENV NODE_ENV=production
RUN addgroup -S nodeapp && adduser -S nodeapp -G nodeapp
COPY --from=deps /app/node_modules ./node_modules
COPY src ./src
USER nodeapp
EXPOSE 3000
CMD ["node", "src/app.js"]

🔑

Moindre privilège

permissions: contents: read global. packages: write et id-token: write isolés au job concerné.

⚡

Cache Node

setup-node avec cache: npm sur les deux lockfiles.

🧬

CodeQL SAST

security-extended, upload SARIF, échec si error ou severity ≥ 7.0.

⛔

Zéro tolérance

continue-on-error: true interdit. Gitleaks, CodeQL ou Trivy en échec = déploiement bloqué.

⛔ Politique d'interruption stricte

Si Gitleaks lève une alerte, si CodeQL trouve une faille majeure (High/Error) dans le backend, ou si le scan d'image échoue, le pipeline s'arrête immédiatement et bloque toute la suite du workflow — donc tout déploiement.

07Composite Action — Trivy SBOM

La gouvernance sécurité est centralisée dans une action locale réutilisable : .github/actions/trivy-scan/.

Sévérité détectéeConséquence pipeline
CRITICALÉchec — --exit-code 1
HIGH / MEDIUMAvertissement dans le résumé du workflow, le pipeline continue
.github/actions/trivy-scan/action.yml
inputs:
  sbom-path:
    description: Path to the CycloneDX JSON SBOM.
    required: true
runs:
  using: composite
  steps:
    - name: Scan CRITICAL vulnerabilities
      shell: bash
      run: trivy sbom --exit-code 1 --severity CRITICAL "${{ inputs.sbom-path }}"

08Déploiement continu

Déclenché uniquement après validation de l'intégralité de la CI et seulement sur un merge validé vers main.

🌐

Front → GitHub Pages (OIDC)

pages: write + id-token: write. upload-pages-artifact package /frontend, deploy-pages propulse le site de façon hermétique, sans trace dans l'historique Git.

▲

Backend → Vercel

Déploiement machine-to-machine via CLI. Secrets déchiffrés par SOPS en RAM et injectés à la volée dans la commande de déploiement.

deploy-frontend — privilèges OIDC isolés
deploy-frontend:
  if: github.ref == 'refs/heads/main'
  needs: [test, gitleaks, codeql, sbom-scan, docker-image]
  environment: production
  permissions:
    contents: read
    pages: write
    id-token: write

09Robustesse, concurrence & rollback

🔄

Concurrence

Deux commits coup sur coup : le pipeline du premier s'annule (cancel-in-progress: true) pour éviter la concurrence lors des déploiements.

🩺

Healthcheck post-déploiement

Un curl de diagnostic frappe /api/health sur l'URL Vercel générée. Tout code ≠ 200 fait échouer le job et notifie l'anomalie.

healthcheck de diagnostic
- name: Healthcheck
  run: curl --fail --silent --show-error "${{ steps.vercel.outputs.url }}/api/health"
